# One-Hour Lab Standard

## Purpose

This document defines the default lab size for the `Mac and AI` track.

Labs should be short enough to complete in one focused session so the learner can complete multiple labs per day.

## Core Rule

Every normal lab should be designed for approximately 60 minutes.

A lab should teach one clear concept, produce one useful piece of evidence, and connect that concept to a real support, operations, engineering, or corporate environment.

Do not write large multi-hour labs unless the lab is explicitly marked as a capstone or review lab.

## Time Box

Use this default structure:

| Time | Activity |
| --- | --- |
| 5 minutes | Read scenario, objective, and constraints |
| 10 minutes | Read or skim official documentation and source notes |
| 30 minutes | Complete the practical or conceptual task |
| 10 minutes | Verify the result and record private notes |
| 5 minutes | Answer reflection questions in private notes |

## Lab Size Limit

A normal lab should usually include:

* one scenario
* one corporate or production context section
* 3-6 requirements
* 3-5 constraints
* 3-5 practical or conceptual tasks
* 3-6 useful commands, checks, or artefacts where relevant
* one verification table
* one short production relevance section
* seven reflection questions

Avoid large lists of commands, long reference tables, long deliverable sections, and too many optional extensions.

## Learning Style

Each lab should still tell a story.

The lab should answer:

* What situation am I in?
* What problem am I solving?
* Why would this matter in a real company?
* What small task proves the concept?
* What should I record in my manual notes?
* What should I avoid doing?

## Scope Control

If a topic is too large for 60 minutes, split it across multiple labs.

Examples:

| Large topic | Split into |
| --- | --- |
| Apple Business Manager | Concept overview, device lifecycle, enrolment workflow, support escalation |
| MDM | Enrolment concept, configuration profiles, restrictions, app deployment, troubleshooting |
| RAG | documents, chunking, embeddings, retrieval, prompt integration, evaluation |
| Cloud deployment | local app, Azure deployment, AWS deployment, GCP deployment, monitoring |

## Evidence Rule

The learner records evidence and reflections privately in manual notes.

The repository lab document should provide the structure, checks, and prompts. It should not require evidence to be submitted back into chat.

## Completion Rule

A one-hour lab is complete when the learner has:

* read the scenario
* completed the main practical or conceptual task
* recorded private notes
* checked the result
* answered the seven reflection questions privately
* avoided unsafe or out-of-scope actions

## Capstone Exception

Capstone and review labs may take longer than one hour, but they should be clearly marked as capstone, review, or multi-session labs.
